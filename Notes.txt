ORM
Relationship b/w tables
One to one field - one item from tbl1 is related to only one item in tbl2
Foreign Key - tbl1 has a column which is unique and this column is also available in tbl2 using this column we can get data from tbl2
Many to Many field - one item from tbl1 can be mapped with multiple items in tbl2 and similarly one item from tbl2 can be mapped with multiple items from tbl1.

on_delete = Cascade (It means that if the parent transaction is deleted then the child will also be deleted.)
E.g.
tbl1 : Order (order_id, order_date)
tbl2: invoice (order_id, items list, price)
so lets say order table is master table and one invoice can have a single order and one order can have a single invoice. What if that one order is deleted from the order table, then the corresponding invoice will also get deleted.

on_delete = Restrict (It means that if you run a query to delete the master transaction then it won't let you delete it, it will ask you to delete the child transaction first then delete the master transaction.)

on_delete = SET_NULL (It will set mapping column id as NULL in child table if you delete the master transaction.)

# Celery
Celery is like CRON job, it is used to schedule tasks at specific time or when any event is triggered to run in background.
e.g. lets say you want to wish your user happy birthday at 12am when they have they day, you can use celery to schedule that.
e.g. lets say you want to download 100's of images or do something which is very CPU intensive, then you cant run that in the main thread of the application so you would like to run this task in the background such that users can do whatever they like in the foreground but in the background it is downloading the files. It makes you app faster and does the job in the background.

Celery has three components to it: Task, Worker, Message Broker (Queue)
Task: It is the task which needs to be performed
Worker: It will perform that task.
Message Broker: It is where the task will be queued, so when it required to run then from here the task will be given to the Worker and then it will get executed. We can use postgres, MySQL or any other database for queueing but they are very slow to work as a message broker so we primarily use "RabbitMQ" or "Redis" as message broker. They are not just fast they are developed for queuing so they have multiple feature complimenting them.

# Celery beat
celery is used to run task in background but beat is used to schedule the task. We can dynamically schedule task using celery beat.

# Celery vs Celery Beat:
Think of Celery as the "worker" and Celery Beat as the "scheduler." They are two separate processes that work together to handle tasks in the background.
To put it in developer terms: Celery Beat is like a crontab that sends a message to the Celery worker queue at specific times.

Webhooks:
A webhook is a lightweight, event-driven HTTP callback that allows one application to automatically send real-time data to another when a specific event occurs. Often called "Reverse APIs," webhooks eliminate the need for constant polling, pushing data instantly via HTTP POST requests to a specified URL.
Example lets say you are using razorpay payment api and when transaction is completed successfully then you can set a webhook which will inform you once th transaction is completed (trigger) using reverse api.



How to use tailwind css in the Django project
1. First you have to download the standalone tailwind file: https://github.com/tailwindlabs/tailwindcss/releases/latest
2. Then move and rename it to a good location like in our case: "C:\tools\tailwind.exe"
3. Then add this path to the environment variable: "C:\tools\"
4. To test whether the env variable is set properly you can run: tailwind --help
5. In the Django project, you have to create a "static" folder in the root folder and add two subfolder css > output.css and src > main.css.
6. To include this static folder in the project you have to include following in the setting file:
STATICFILES_DIRS = [
    BASE_DIR / "static",
]
7. Since the standalone file size is 125mb, you have to run a command so that we only include required components form the main file into the output.css file: tailwind -i ./static/src/input.css -o ./static/css/output.css --watch
While you are changing anything in the css file everything gets recorded in the output.css file. so during the deployment you will only have to upload less than 1 mb file as css file for using tailwind css components.
Remember to keep this command running while you are working on the project as any new changes is recorded in the output file at the time of dev. so you will need two terminals while dev, first will run the server and the other will run this command.
8. once you are done with the project and you want to deploy it just remove "--watch" from above command to make it the final deployable file.

API Methods or HTTP Methods
GET: Fetch data from backend to frontend/client.
POST: Create a new entry in backend from client/frontend.
PUT: When you want to update the existing data in the backend from client/frontend.
PATCH: When you want to partially update the data in the backend from client/frontend.
DELETE: Delete any entry in the backend from client/frontend.


DRF Serializer:
Converts data from one format to another: Here it converts queryset to JSON. Also it inserts data from frontend to DB.
Serializer is of multiple type:
1. Basic Serializer: You have to build everything from scratch (fields, method).
2. Model Serializer: It connects to any model and all the fields from model is copied into the Serializer.
3. Hyperlink Serializer: In case of any foreign key, instead of returning the value, it return hyperlink to access the value.
4. List Serializer: It uses list to create Serializer.
