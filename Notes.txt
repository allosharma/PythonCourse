ORM
Relationship b/w tables
One to one field - one item from tbl1 is related to only one item in tbl2
Foreign Key - tbl1 has a column which is unique and this column is also available in tbl2 using this column we can get data from tbl2
Many to Many field - one item from tbl1 can be mapped with multiple items in tbl2 and similarly one item from tbl2 can be mapped with multiple items from tbl1.

on_delete = Cascade (It means that if the parent transaction is deleted then the child will also be deleted.)
E.g.
tbl1 : Order (order_id, order_date)
tbl2: invoice (order_id, items list, price)
so lets say order table is master table and one invoice can have a single order and one order can have a single invoice. What if that one order is deleted from the order table, then the corresponding invoice will also get deleted.

on_delete = Restrict (It means that if you run a query to delete the master transaction then it won't let you delete it, it will ask you to delete the child transaction first then delete the master transaction.)

on_delete = SET_NULL (It will set mapping column id as NULL in child table if you delete the master transaction.)

# Celery
Celery is like CRON job, it is used to schedule tasks at specific time or when any event is triggered to run in background.
e.g. lets say you want to wish your user happy birthday at 12am when they have they day, you can use celery to schedule that.
e.g. lets say you want to download 100's of images or do something which is very CPU intensive, then you cant run that in the main thread of the application so you would like to run this task in the background such that users can do whatever they like in the foreground but in the background it is downloading the files. It makes you app faster and does the job in the background.

Celery has three components to it: Task, Worker, Message Broker (Queue)
Task: It is the task which needs to be performed
Worker: It will perform that task.
Message Broker: It is where the task will be queued, so when it required to run then from here the task will be given to the Worker and then it will get executed. We can use postgres, MySQL or any other database for queueing but they are very slow to work as a message broker so we primarily use "RabbitMQ" or "Redis" as message broker. They are not just fast they are developed for queuing so they have multiple feature complimenting them.

# Celery beat
celery is used to run task in background but beat is used to schedule the task. We can dynamically schedule task using celery beat.

# Celery vs Celery Beat:
Think of Celery as the "worker" and Celery Beat as the "scheduler." They are two separate processes that work together to handle tasks in the background.
To put it in developer terms: Celery Beat is like a crontab that sends a message to the Celery worker queue at specific times.

# HTTP connection - stateless, once request response cycle is done then there is no relationship b/w server and client. If client has another request then it will send a new request and will receive new response independent of previous connections. WSGI (Gunicorn) can be used to deploy stateless connection projects.
# Web Sockets: Client and server stay connected once the connection is made, so new request does not need a new connection, it can use existing connection. It's called stateful connection.
# In Django to use web sockets we make use of Channels
	Django Channels uses asynchronous connection, so we can deploy it only using ASGI (Daphini), WSGI is not useful in this case.
	When we are using web socket then our request line so we will need message broker, we can either use RabbitMQ or Redis for queuing the messages. To add URL, we can either create a new urls.py file and then call it in ASGI or just add the URLS's in the ASGI file itself.
	urls.py and views.py which we use for WSGI won't work for ASGI.
